# AirPad Codebase Audit — 2026-04-30

HEAD: 9962351
Auditor: Claude Code (Opus 4.7, read-only)
Scope: read-only investigation, no code changes

## Summary

Codebase is broadly healthy and the SB80→SB85 evolution is legible from comments and state. Three things stand out before SB91 lands. **(1)** A whole zoom apparatus (`centerAndZoomNode` + ~6 backing fields) is dead — never invoked from any caller — but its sibling `resetZoom` is still wired to UI taps and silently no-ops. **(2)** Two of the four phase-1 SB85 lens primitives are still defined but have no callers in the SB91 layout pipeline (sigmoid `screenFractionForNormalizedDistance` and constants `scaleSigmoidSteepness` / `scaleSigmoidMidpoint` / `baselineScreenFraction` / `neighborBreathingGap`). The lens math is half SB85, half SB91, with the SB85 sigmoid path now an island. **(3)** The thrashing the user is seeing during drag is structurally consistent with the SB91 slot-ring design: discrete band/ring boundaries + sort-order slot assignment + steady-state lerp at 0.20 + 1-frame focal-switch jump = visible per-node target leaps on every focal switch. SB91 will likely worsen this unless slot continuity is added (Section 7E + Recs D).

Nothing here would prevent SB91 from compiling and running. Bucket A is empty. Bucket B (cheap and obvious) covers ~40 lines of dead code that would make the next dispatch easier to read. Bucket D (transition behavior) is the one that materially matters for the next change.

## 2. Engagement state machine

States (CorpusPhysicsScene.swift:495):
```
idle
engaging(focal: String)
engaged(focal: String)
gracePeriod(focal: String, expiresAt: TimeInterval)
disengaging
```

Transition diagram:

```
                       ┌────────────────────────────────────────────┐
                       │                                            │
                       ▼                                            │
[idle] ── tapCandidate→honeycomb (drag>10pt, not in grace) ──▶ [engaging(focal)]
   ▲                                                              │  │
   │                                                              │  │ same-focal frame loop
   │                                                              │  │ converges (Phase 3)
   │                                                              │  ▼
   │                                            ┌──── focal switch (find-nearest+hysteresis) ────┐
   │                                            │                                                │
   │                                            │                                                ▼
   │                                            │                                          [engaged(focal)]
   │                                            ▼                                                │
   │                                  [engaging(newFocal)]  ◀─ self-loop on focal switch ─┐      │
   │                                            │                                          │     │
   │                                            ▼                                          │     │
   │                                            (carries on; loops back to convergence)    │     │
   │                                                                                       │     │
   │                                                                                       │     │
   │                                  ┌──── engagement.engaged ── self-loop (focal switch) ┘     │
   │                                  │                                                          │
   │                                  ▼                                                          │
   │                          touchesEnded:                                                      │
   │                          if currentFocalNodeID != nil ──▶ [gracePeriod(focal, expiresAt)] ◀─┘
   │                                  │
   │                                  │ touchesBegan in grace (tap on a node)
   │                                  │   ─▶ refresh expiresAt (stay in grace)
   │                                  │
   │                                  │ update(): currentTime ≥ expiresAt
   │                                  │   ─▶ [disengaging]
   │                                  │
   │                                  │ touchesMoved during grace, drag>10pt
   │                                  │   ─▶ [disengaging]
   │                                  ▼
   │                          touchesEnded with no currentFocalNodeID  ─▶ [disengaging]
   │                          touchesCancelled                         ─▶ [disengaging]
   │                                  │
   │                                  ▼
   └──── update(): all sprites converged to nodeRestingPositions ─── [idle]
```

Findings on the machine:

- **All five states are reachable.** idle is the start state; engaging and engaged are reached via the drag-from-tapCandidate path; gracePeriod is reached from touchesEnded with a focal; disengaging is reached from gracePeriod expiry / drag-during-grace / touchesEnded-without-focal / touchesCancelled.
- **No defined-but-unreachable transitions.**
- **Two paths to `disengaging` from `gracePeriod` overlap but don't conflict:** the time-based expiry (update() L990) and the drag-during-grace path (touchesMoved L2332). On the same frame they can't both fire — the touch-handler runs before the next update tick. Both target the same state, so there's no race in observable behavior.
- **Three paths reach `engaged` from `engaging`:** the convergence-based transition (L962) and two self-loops on focal switch (L702-L706 inside the gestureState handler). The self-loops effectively keep the state name but swap focal — this is intentional but worth flagging because they sidestep the "must converge before engaged" gate. A fast focal-switch series can ratchet through `engaging(A)→engaged(A)→engaged(B)→engaged(C)` without ever fully converging on (B) — see Section 7C.
- **Hold-timer vs. grace-timer ambiguity:** `holdTimerStart` (line 488) gates the strands+drift render at L720 (>= 0.7s same-focal hold). `holdTimerStart = currentTime` is reset on every focal switch (L709). It's only consulted while `gestureState` is `.honeycomb` AND `engagementState` is in active engagement. Grace-period timer lives on `engagementState.gracePeriod` only. They never overlap because `gestureState` returns to `.idle` on touchesEnded before grace begins. **No genuine ambiguity** — but `holdCompleted` flag is written and never read (Section 1.4), suggesting an older design where the two timers did interact and got cleaned up partially.
- **`engagementState` is mutated from both touch handlers and update().** Both run on the main thread (touch via UIKit, update() via SKScene tick), so this is sequential, not concurrent. Safe.

## 4. Gesture flow

Touch-to-engagement-state-change pipeline:

1. **`touchesBegan`** (L2227)
   - Kill momentum, clear `panSamples`, `momentumEligible = false`.
   - Record into `activeTouches`.
   - **If 1 finger AND `engagementState == .gracePeriod`:** hit-test `nodeSprites.values.first { $0.contains(scenePoint) }`. If hit a node:
     - Check `(now − lastGraceTapTime) < doubleTapWindow (0.35s)` AND same node → set `canvasState.pendingNavigationNodeID = tappedNodeID` (push detail).
     - Else arm `lastGraceTapNodeID`, refresh `expiresAt = now + gracePeriodDuration (2.0s)`. Stay in grace.
     - Return.
   - Else (1 finger, not in grace): `gestureState = .tapCandidate(initialPosition, startTime)`.
   - **If 2 fingers:** record pinch distance, cancel tap (`tapStartInfo = nil`), `gestureState = .idle`.

2. **`touchesMoved`** (L2297)
   - **In `.tapCandidate` AND drag distance > `dragThreshold (10pt)`:**
     - If `engagementState == .gracePeriod`: clean up grace (clear strands, unwind drift, restore z-order, fade prompt label) and **set `engagementState = .disengaging`** (L2332). `currentFocalNodeID = nil`. The drag itself becomes a fresh pan.
     - Else (was idle): `focalID = findNearestNodeToCamera() ?? ""`; **set `engagementState = .engaging(focal: focalID)`** (L2337).
     - Either way: `gestureState = .honeycomb(initialPosition, current)`; `holdTimerStart = now`; `momentumEligible = true`.
   - **In `.honeycomb`:** pan camera by `(current − lastPanPosition) * panMultiplier (1.5) * cameraNode.xScale`, append to `panSamples` ring buffer (100ms window).
   - **2 fingers:** pinch zoom (clamped 0.25…4.0).

3. **`update(_:)`** per-frame (L625)
   - Coast-camera momentum (if `coastVelocity != .zero`).
   - Update label tier from `cameraScale`.
   - Update halos.
   - Update strand paths (if `focalNodeID` is set — note that's distinct from `currentFocalNodeID`).
   - **Honeycomb focal-tracking branch** (`gestureState == .honeycomb` AND `isInActiveEngagement`):
     - `newFocalID = findNearestNodeToCamera()` (with 20pt hysteresis).
     - If different from `currentFocalNodeID`: restore old focal's z, raise new focal's z, **mutate engagementState's focal** while keeping the same case (`engaging` or `engaged`), reset `holdTimerStart`, clear strands, unwind drift.
     - If same focal AND `(now − holdTimerStart) >= holdThreshold (0.7s)`: render strands, start drift, set `holdCompleted = true` (write-only flag, see Section 1.4).
   - **Engagement state branch:**
     - `.engaging` / `.engaged`: SB91 Phase 1A–1E (partition by lens band → assign slot ring → compress periphery), Phase 2 (bounded-band relaxation, max 5 passes), Phase 3 (lerp). Convergence → `engaging` upgrades to `engaged`. Camera follow only in `.engaged`.
     - `.gracePeriod`: wait for `expiresAt`. On expiry: clean up, transition to `.disengaging`.
     - `.disengaging`: lerp toward `nodeRestingPositions`/`Scales`. On convergence: `.idle`.

4. **`touchesEnded`** (L2397)
   - **In `.honeycomb`:** maybe launch `coastVelocity` from `panSamples` (if `momentumEligible` and last-vs-first sample velocity ≥ `coastLaunchThreshold (2.0px/frame)`). If `currentFocalNodeID != nil`: **`engagementState = .gracePeriod(focal, now + 2.0s)`**, reset `gestureState = .idle`, fade-in "Tap for more detail" label. Else: **`engagementState = .disengaging`**.
   - **Otherwise (single tap path)** — only if this is the last touch lifting AND duration < 0.3s AND distance < 14pt:
     - Hit-test scene point. If hit a node sprite: `canvasState.selectedNodeID = nodeID` (opens NodeDetailOverlay).
     - Else if double-tap on empty space AND `drilledInto != nil`: clear drill-down.
     - Else if `zoomedNodeID != nil`: `resetZoom()` (always a no-op given `zoomedNodeID` is never set — see Section 1.1).
     - Else: clear `selectedNodeID`.

5. **`touchesCancelled`** (L2523)
   - Clean up everything; `engagementState = .disengaging`; restore focal scale and z; `gestureState = .idle`.

### Findings

- **Hit-test for tap-on-node happens in two places** — `touchesBegan` during `.gracePeriod` (L2251) and `touchesEnded` for normal taps (L2494). They use different state machines (touch-down vs. touch-up) and different double-tap windows (0.35s vs. 0.3s). Not double-handling the same event, but the inconsistent windows (0.35s grace vs 0.3s normal) are easy to overlook.
- **`SKShapeNode.contains(_:)` against blob-deformed paths.** Both hit-test sites walk `nodeSprites.values.first { $0.contains(scenePoint) }`. Sprites are continuously remorphed by `startBlobBreathing` (L1926). At the moment of touch the bounding hit can include the deformed path's most-recent shape — so a touch right at the edge of a node may catch or miss depending on which lobe is currently bulging out. Likely fine in practice (deformation amplitude is 0.15) but worth knowing.
- **`engagementState` self-loops on focal switch sidestep the convergence gate.** A rapid focal-switch sequence in `.engaged` keeps state at `.engaged` and lerp at the steady-state factor (0.20), even though no individual focal target ever fully converges. This is the structural source of mid-drag thrashing — see Section 7.
- **Hardcoded inline values that are not declared as named constants:**
  - `0.3` — touch-tap max duration (L2479).
  - `14` — touch-tap max travel distance (L2479).
  - `0.3` — empty-space double-tap window (L2488).
  - `30` — empty-space double-tap distance threshold (L2488).
  - `0.08` — drift-toward-focal fraction (L1375–1376).
  - `0.5` / `0.4` — drift animation durations (L1379, L1392).
  - `60.0` — characteristicSpacing fallback (L1089, L1107).
  - `5` — drift-related-nodes limit (L733, L1262).
  - `0.55` — rearrange animation duration (L23).
  - `0.45` / `0.38` — various sprite animation durations across `addNodeSprite`, `centerAndZoomNode`, `resetZoom`.
  - `60` — y-offset for "Tap for more detail" label (L2449).
- **`lastTapTime` is updated even when the tap hits a node** (L2491). Subsequent double-tap-on-empty detection uses it. Not buggy in current paths (the node-tap path opens detail and the empty path checks `isDoubleTap` against the now-tainted `lastTapTime`), but counterintuitive.
- **`focalNodeID` (strand layer, L240) vs. `currentFocalNodeID` (gesture, L487).** Two separately-tracked focal IDs that happen to refer to the same concept. `setFocalNode` writes one; the gesture machine writes the other. They get aligned via `setFocalNode(focalID)` at L725, but a future edit could easily break the relationship. Not a current bug.

## 7. Transition behavior during drag

### 7A. Target recomputation cadence

- **Phase 1 of the engagement loop runs every frame** during `.engaging` or `.engaged` (CorpusPhysicsScene.swift:751–878). `targetPositions` and `targetScales` are declared **fresh inside the case body** (L771–772) — there is no per-frame carry-forward of last frame's targets.
- **On a focal switch mid-drag**, the gestureState focal-tracking branch (L702–706) mutates `engagementState`'s focal (`engaged(focal: oldID)` → `engaged(focal: newFocalID)`), and the **next** frame's lens math reads the new focal. All targets recompute from scratch around the new focal. There is **zero continuity** preserved from the previous focal's targets — the partition (in-disc / inner periphery / far periphery) is redone, the slot ring is rebuilt from the new focal as anchor, and the inner ring's starting angle (`firstAngle = inDiscNodes[0].angle`, L821) is sampled from a different node than before.
- **No per-node memory of prior heading.** Each sprite's target on frame N+1 is a pure function of the new focal and the canonical resting fingerprint — it doesn't know what target it was lerping toward on frame N.

### 7B. Lerp behavior during transitions

- `engagementLerp = 0.12` (L557) → used during `.engaging`.
- `steadyStateLerp = 0.20` (L558) → used during `.engaged`.
- Switchover is at the `engaging → engaged` transition (L962–965), gated on `allPositionsConverged && allScalesConverged`.
- **A mid-drag focal switch does NOT change the lerp factor**, because the engagementState case label stays the same (`.engaged → .engaged(newFocal)`). So if the user is in `.engaged` with focal A and pans into focal B's basin, sprites that were comfortably lerping at 0.20 toward A's slot ring suddenly have B's slot-ring positions as their target — and they lerp toward those new targets at the same fast 0.20 factor. ~80% of the gap closes in ~8 frames.
- During `.engaging`, the slower 0.12 factor smooths a focal switch better — but per the L962 gate, the engaging→engaged transition fires the moment everything converges, so during a smooth slow drag the system gets to `.engaged` quickly and then stays there for the rest of the drag.

### 7C. Convergence and target stability

- `allPositionsConverged` / `allScalesConverged` (L923–924, L997) are per-frame booleans that aggregate over all sprites, gated by `positionMatchTolerance = 2.0` (L562) and `scaleMatchTolerance = 0.05` (L563).
- The **only state-machine consumer** of these flags is the `.engaging → .engaged` transition (L962) and the `.disengaging → .idle` transition (L1031). They do **not** gate focal switches in `.engaged`. So a sprite mid-ease toward target T1 of focal A can have its target swapped to T2 of focal B, and the convergence flag simply flips back to false until T2 is reached. If focal switches again before T2, target becomes T3, and so on — the sprite never catches up.
- This is the central thrashing mechanism: **no convergence gate on focal switches in `.engaged`, combined with discrete target recomputation, combined with fast steady-state lerp**.
- No nodes have intentionally non-stable targets. Drift (L1360, `startDrift`) uses `SKAction.moveBy` independently of the lens loop and excludes drifting nodes via `driftExcludedIDs` (L530), so drift is not contributing to thrashing.

### 7D. Focal switching threshold

- **What gates a focal switch:** `findNearestNodeToCamera()` (L1321) — picks the sprite nearest to **`cameraNode.position`** (the camera center, NOT the touch point). During honeycomb pan, the camera moves opposite the drag direction, so this is effectively "which node is closest to the screen center."
- **Hysteresis:** `hysteresisThreshold = 20.0` (L565). New candidate must be `at least 20pt closer` to camera-center than the current focal before switching (L1346). **20pt is not very large** at default zoom, especially given `characteristicSpacing` typically lands in the 60–120pt range — the user can cross the hysteresis boundary every few node-widths of drag.
- **The 20pt is also unitless world points, not screen pixels**, so at zoomed-out (`cameraScale > 1`) the threshold becomes effectively smaller in screen-space.
- **Focal switching does NOT directly call any layout-recompute logic.** L702–706 only update `currentFocalNodeID` and the engagementState's focal. The recompute happens implicitly on the next frame via the engagementState case branch.

### 7E. Thrashing reproducibility (analytic prediction)

For a slow horizontal drag across 5 adjacent nodes, the predicted behavior is **(b) discrete jumps at each focal-switch boundary, smoothed by the lerp**.

Mechanisms by which a non-focal node's target can change by more than ~30% of `characteristicSpacing` in a single frame:

1. **In-disc → inner-periphery boundary crossing.** A node sitting at world distance `lensDiscWorldRadius − ε` (just inside the disc) when focal=A may end up at world distance `lensDiscWorldRadius + ε` (just outside) when focal=B. Its target instantly switches from "slot on the inner ring" (a position roughly `focalTargetWorldRadius + slotRingBreathingGap + neighborEdgeWorldRadius` away from focal) to "compressed periphery" (resting position, slightly pushed). For typical screen sizes this difference is several `characteristicSpacing` units. (CorpusPhysicsScene.swift:799–807 + L857–866.)
2. **Slot reassignment within the in-disc set.** Slot indices are assigned in **sorted angular order around focal** (L809), and the inner ring uses `firstAngle = inDiscNodes[0].angle` as the offset (L821). On focal switch, both the angle of every node and the identity of "inDiscNodes[0]" change — so most in-disc nodes get assigned to a different slot. Adjacent slots are `2π / innerCount` radians apart, which is usually one full neighbor-diameter at the inner ring's radius. Visible jump.
3. **Inner-ring vs. outer-ring promotion/demotion.** With `maxNodesInInnerRing = 8`, the 9th node in sorted angular order goes to the outer ring at `outerRingRadius` (~1 inner-diameter farther out). Whether a particular node is the 8th or 9th is order-dependent and changes on focal switch — so a node can pop between rings with no smooth interpolation.
4. **Bounded-band relaxation order-dependence** (L888–920). The relaxation iterates pairs `(i, j)` in the order of `relaxationSet` — which is in-disc-by-angle then inner-periphery in dictionary-iteration order (L883–886). Iteration order changes when focal changes (because `inDiscNodes` is rebuilt) and when set membership changes. The same overlap can resolve to different final positions depending on order. Up to `relaxationPasses = 5` iterations, then it bails. Visible "settle" wobble.

All four conditions are present in the SB91 path that ships at HEAD `9962351`. Expect visible per-frame leaps, then a fast (0.20-lerp) rush toward the new arrangement.

### D. Transition behavior — implications for SB91

**Will SB91's slot-ring approach improve, worsen, or leave unchanged the thrashing the user is seeing during drag?**

**Worsen, in expectation, unless slot continuity is added.** Reasoning:

- SB85 (the `.engaging`/`.engaged` lens at, e.g., commit `cab0f93`) was a **continuous** function of distance: each non-focal node's target was a smooth radial-compression position + sigmoid scale. Small focal-position changes → small target changes per node → smooth per-frame motion.
- SB91 carves the layout into **discrete bands** (in-disc / inner periphery / far periphery) and **discrete slots** (8-cap inner ring, outer ring at a different radius). Boundary crossings on focal switch produce per-node position jumps of ≥1 neighbor-diameter (Section 7E).
- The same `engagementLerp` / `steadyStateLerp` factors smooth those jumps, but the magnitude of the discrete jumps is larger than what SB85 produced.

**What SB91 needs to consider, if the user wants to keep the slot-ring shape:**

1. **Persistent slot assignment across focal switches.** Right now, `inDiscNodes.sort { $0.angle < $1.angle }` (L809) re-assigns by sorted angle every time focal changes. If you keep an `[String: Int]` mapping of `nodeID → slotIndex` (with TTL on focal-change so a node doesn't migrate in mid-cycle), a node that was in slot 3 around focal A stays close to slot 3 around focal B. That's the SB79-honeycomb-v10 idea ("watchOS-style continuity") — applied to slot ring instead of honeycomb.
2. **Smoothly interpolate band-membership transitions.** When a node crosses `lensDiscWorldRadius`, blend the slot-ring target and the radial-compression target for a few frames rather than snap. Equivalent to a soft membership function instead of a hard partition.
3. **Larger / scaled `hysteresisThreshold`.** 20pt of world-space at typical zoom-out becomes <10pt of screen-space. Scale the threshold by `1.0 / cameraScale` so the focal-switch frequency stays consistent across zoom levels. Currently the focal switches more often when the canvas is zoomed out, which compounds the thrashing.
4. **Gate steady-state lerp on focal stability.** Drop to `engagementLerp` (0.12) for ~10 frames after every focal switch, then ramp back to `steadyStateLerp` once positions are within tolerance. Trades convergence speed for visual smoothness right where the user perceives the thrashing.

Of these, (1) gives the largest quality improvement for the smallest design change. (3) is essentially a one-line scale-aware tweak. (4) is small and could be implemented without touching the slot-ring math at all.
